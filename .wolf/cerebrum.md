## Key Learnings

- **Project:** claude
- App project renamed from `BlinddensportGraz` to `BlindensportGraz` (2026-07-15, typo fix: double "d" removed). Bundle ID is now `it.a11y.BlindensportGraz`. `BlindensportGraz.xcodeproj/project.pbxproj` is fully generated by XcodeGen from `project.yml` — never hand-edit the pbxproj; edit `project.yml` and run `xcodegen generate`.
- An "Apple Development" codesigning cert's CN shows a per-certificate identifier in parentheses (e.g. `Apple Development: Name (V756YZC78P)`), which is NOT necessarily the Team ID. The real Team ID is in the cert's OU field, or read via `defaults read com.apple.dt.Xcode IDEProvisioningTeams` / `IDEProvisioningTeamByIdentifier`. Don't assume the parenthetical value is the team.
- First-time device deployment: `xcodebuild -allowProvisioningUpdates` from a pure CLI/background session frequently fails to auto-register a brand-new device ("isn't registered in your developer account") even with a valid paid-account cert. Opening the project in Xcode GUI and hitting Run once with that device selected completes the interactive registration; CLI builds work for that device afterward.
- Sign in with Apple only ever sends real name/email on the genuine first-ever grant for an Apple ID + Bundle ID/Team combo — this is tracked at the OS/Keychain level, NOT reset by uninstalling the app, clearing local UserDefaults/SwiftData, or even by the app never appearing in Settings > Sign-In & Security > Apps Using Apple ID. Once withheld, the only user-side reset is fully signing out of the Apple ID on the device and back in. Production-correct handling (already implemented in RootView.resolveAccount) is to accept whatever Apple provides and let the user fill in the rest via a manual profile-edit screen — don't expect or debug toward "it should give real data every fresh install," that premise is wrong.

- [2026-07-15] fastlane's `sh("command", chdir: DIR)` does not reliably change directory before the command resolves its own relative-path lookups — `xcodegen generate` still searched `fastlane/` for `project.yml` instead of `DIR`. Fix: use `sh("cd '#{DIR}' && command")` instead of the `chdir:` keyword arg.
- [2026-07-15] Do NOT keep retrying `xcodebuild`/`fastlane` codesigning from this coding session's Bash tool once `errSecInternalComponent` + "User interaction is not allowed" (from `security show-keychain-info`) appear together. This is a sandbox-level block on reaching the Security Agent, not a keychain config problem — running `security set-key-partition-list -S apple-tool:,apple:,codesign:` (even with "Always Allow") does NOT fix it, confirmed by direct retest. Stop and use the Xcode-GUI-via-osascript workaround (see bug-008) instead of iterating on keychain/ACL changes.
- [2026-07-15] The osascript Cmd+R trick to trigger Xcode builds can silently stall: if the app is already running on the device, Xcode shows a "Replace 'X'? An instance is already running..." modal sheet and Cmd+R does nothing further until it's dismissed. After sending Cmd+R, check for a sheet with `osascript -e 'tell application "System Events" to tell process "Xcode" to count of sheets of window 1'` and if present, click through it (e.g. `click button "Replace" of sheet 1 of window 1`) before assuming the build is progressing. Verify actual success via `xcrun devicectl device info processes --device <id> | grep <AppName>` — a changed PID/bundle-container UUID confirms a fresh install+launch, not just the old still-running process.
- [2026-07-16] In a Swift Package executable target, a source file literally named `main.swift` is compiled as implicit top-level-code script and conflicts with `@main` on a struct/class ("'main' attribute cannot be used in a module that contains top-level code") — SPM's special-casing of that exact filename, not a Swift language rule. Fix: name the `@main`-bearing file anything else (e.g. `RootCLI.swift`); `main.swift` only if you're writing top-level script code directly with no `@main` type.

- [2026-07-16] Added the "Grazer VSC" club-membership feature. `ClubMember` (Models.swift) is the admin-managed roster (fullName/address/email/phone/memberNumber/joinedAt/notes), synced cross-device via `CloudKitSync.push/pullClubMember` on the same public-database pattern as Team/Event/etc — matches the existing "no built-in SwiftData CloudKit for cross-user sharing" convention (see Decision Log below). `User.isGrazerVSCMember` is a Bool flag auto-computed by `ClubMember.checkMembership(for:modelContext:)`, called at both account-creation sites (RootView's Apple Sign-In auto-create path and RegisterView's manual path) — not re-checked on every login, only at creation, per the requirement's literal wording ("when a new user account is created"). Matching (`ClubMember.matches`) is case/whitespace-insensitive, email-first then falls back to exact displayName match. The flag is published in `pushUserIdentity`/`pullUserIdentities` alongside role, so it's visible across devices like other identity fields.

- [2026-07-16] Built the root/superuser role-escalation feature (user asked "super user that can change role to admin, user itself can not set his role"). `User.isRoot: Bool` (Models.swift) is a flag distinct from `role`, bootstrapped `true` (plus `role = "admin"`) only on the very first account ever created — checked via `CloudKitSync.hasAnyUserIdentity()` (queries the public DB for any existing UserIdentity before minting a new root) so two devices signing up offline-first don't both self-elect. `EditAccountView`'s role Picker was removed entirely (no self-service, not even for root); role editing moved to `UserListView`, gated on `currentUser.isRoot` and disabled on the root's own row. `xcrun cktool` (bundled with Xcode) was investigated as an alternative to hand-rolled CloudKit REST calls but has no record-update/modify subcommand (only create/query/delete/schema-import-export) — insufficient for flipping an existing user's role, so a custom CLI was built instead. cktool remains useful for the one-time schema/Security-Roles setup via export-schema/import-schema (documented in RootCLI/README.md as an alternative to manual Dashboard clicking).
- [2026-07-16] Added photo uploads for SportEvent/Training/Tournament. `EventImage` (Models.swift) has `@Attribute(.externalStorage) var imageData: Data` plus three optional back-refs (`event`/`training`/`tournament` — exactly one set per image, mirroring the existing "one owner of three possible" pattern used nowhere else yet in this codebase, so future similar cases should follow this shape rather than inventing a polymorphic base). CloudKit sync uses `CKAsset` (a temp file staged then cleaned up in a `defer`), not a raw Data field — the established push/pull pattern for every other model uses plain CKRecord fields, so `pushEventImage`/`pullEventImages` in CloudKitSync.swift are the first exception, worth checking as the reference if another asset-heavy feature comes up. Images are treated as immutable once uploaded — `pullEventImages` skips any id already present locally rather than re-downloading the asset, since CKAsset downloads aren't free. `EventImagesSection` (new file EventImagesViews.swift) is a single reusable SwiftUI view embedded identically in all three detail views (EventDetailView/TrainingDetailView/TournamentDetailView) — it takes a resolved `[EventImage]` array plus add/delete closures rather than being generic over the model type, since SwiftData relationships aren't polymorphic. "Randomly shown on open" is implemented as `@State private var featured: EventImage?` picked via `.onAppear`, which naturally re-randomizes because SwiftUI creates a fresh view instance each time a NavigationLink pushes the detail screen. Picked photos are downscaled to max 1600px + JPEG-compressed (`ImageProcessing.downscaledJPEGData`) before ever touching SwiftData/CloudKit, to keep local storage and CKAsset upload size reasonable. Upload is open to any signed-in user (matches real-world "attendees share photos" usage); delete is restricted to the uploader or an admin.

## Decision Log

<!-- Significant technical decisions with rationale. Why X was chosen over Y. -->

- [2026-07-15] SUPERSEDED (see below): private CloudKit sync for User was reverted. SwiftData's `ModelConfiguration.cloudKitDatabase` only supports the PRIVATE database (syncs one Apple ID's own devices) — there is no `.public` case. It can never satisfy "shared with all users/teammates" since that needs cross-Apple-ID visibility. Don't reach for SwiftData's built-in CloudKit integration for any cross-user sharing feature; it's private-only.
- [2026-07-15] Cross-user, team-scoped sharing (events/trainings/tournaments visible to all members of a team, across different Apple IDs) is implemented via a hand-written `CloudKitSync.swift` using raw CloudKit (`CKContainer.publicCloudDatabase`), NOT SwiftData's automatic mirroring. Local SwiftData is the UI's source of truth (plain local store, `cloudKitDatabase: .none`); CloudKitSync pushes a CKRecord after every create (matching CKRecord.ID.recordName to the local `id.uuidString` for idempotent upsert) and pulls+merges via `syncAll(modelContext:)` on login and pull-to-refresh. Only non-sensitive User fields (username/displayName/role) are ever published as "UserIdentity" records — email and appleUserIdentifier never leave the device. Team membership records must be pulled before Events/Trainings/Tournaments/Participations so relationship lookups by id resolve on first sync (see pull ordering in `CloudKitSync.syncAll`).
- [2026-07-15] Since relationships are no longer forced optional by SwiftData-CloudKit (custom sync doesn't have that constraint), `TeamMembership.user/team` and `EventParticipation.user/event` are back to non-optional, and `@Attribute(.unique)` is back on every `id` — this was an explicit user correction ("a relationship is only possible with a registered user account"). Don't make these Optional again unless re-enabling SwiftData's own CloudKit mirroring specifically.
- [2026-07-15] `SportEvent` and `Tournament` support MULTIPLE teams (`teams: [Team] = []`, many-to-many) since an event/tournament can involve several teams competing; `Training` stays single-team (`team: Team?`) since a training session belongs to one team. Empty/nil = visible to everyone (matches the original "seen by every user" default when no team is assigned).
- [2026-07-15] Found and fixed a significant pre-existing bug (present since before this session started, not introduced by any of the above changes): `BlindensportGrazApp.swift` built a custom `ModelContainer` in `init()` but never attached it to the view hierarchy via `.modelContainer(modelContainer)` on the `WindowGroup`. This caused data created in one view (e.g. a new Team) to not reliably appear elsewhere (e.g. TeamsListView). Confirmed fixed for Teams, Events, Trainings, and Tournaments after adding the modifier. If any "created X doesn't show up elsewhere" bug resurfaces, check this attachment first before suspecting the specific feature's code.
- [2026-07-15] Localization added: German is the source language (`developmentLanguage: de` in project.yml), English translations live in `BlindensportGraz/Localizable.xcstrings` (95 keys, hand-authored but verified against Xcode's own build-time extraction — compiled the app and diffed `en.lproj/Localizable.strings`/`de.lproj/Localizable.strings` to confirm every hand-written key, including interpolated ones like `"Mitglieder (%lld)"` and `"Willkommen, %@"`, matched exactly). Key gotcha: `Text(someString)` where `someString: String` (not `LocalizedStringKey`) NEVER localizes, even with a matching catalog entry — this silently broke `AccountView.roleLabel(_:)` (returned `String`), `DashboardView`'s `StatCard.title`/`sectionHeader(_:)` (both typed `String`). Fixed by changing those to return/accept `LocalizedStringKey` instead. When adding new UI text anywhere in this app, only literal strings passed directly to `Text`/`Label`/`.navigationTitle`/etc. (or through a param typed `LocalizedStringKey`) get localized — passing a pre-built `String` breaks it silently with no compiler warning.
