- An "Apple Development" codesigning cert's CN shows a per-certificate identifier in parentheses (e.g. `Apple Development: Name (V756YZC78P)`), which is NOT necessarily the Team ID. The real Team ID is in the cert's OU field, or read via `defaults read com.apple.dt.Xcode IDEProvisioningTeams` / `IDEProvisioningTeamByIdentifier`. Don't assume the parenthetical value is the team.
- First-time device deployment: `xcodebuild -allowProvisioningUpdates` from a pure CLI/background session frequently fails to auto-register a brand-new device ("isn't registered in your developer account") even with a valid paid-account cert. - [2026-07-15] The osascript Cmd+R trick to trigger Xcode builds can silently stall: if the app is already running on the device, Xcode shows a "Replace 'X'? - [2026-07-16] In a Swift Package executable target, a source file literally named `main.swift` is compiled as implicit top-level-code script and conflicts with `@main` on a struct/class ("'main' attribute cannot be used in a module that contains top-level code") ŌĆö SPM's special-casing of that exact filename, not a Swift language rule. Fix: name the `@main`-bearing file anything else (e.g. `RootCLI.swift`); `main.swift` only if you're writing top-level script code directly with no `@main` type.

- [2026-07-15] SUPERSEDED (see below): private CloudKit sync for User was reverted. SwiftData's `ModelConfiguration.cloudKitDatabase` only supports the PRIVATE database (syncs one Apple ID's own devices) ŌĆö there is no `.public` case. - [2026-07-15] Cross-user, team-scoped sharing (events/trainings/tournaments visible to all members of a team, across different Apple IDs) is implemented via a hand-written `CloudKitSync.swift` using raw CloudKit (`CKContainer.publicCloudDatabase`), NOT SwiftData's automatic mirroring.
